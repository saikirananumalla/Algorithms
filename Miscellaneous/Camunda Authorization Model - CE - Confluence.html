Understanding Camunda Authorization Model

  • By default Authentication and Authorization are disabled in embedded Camunda running with Spring-boot.

  • We can enable Basic Authentication using Basic Auth (provided with Camunda) Or with custom Authentication like admin-svc. In case we enable Authentication using admin-svc, all Camunda REST API will be secured and need a valid JWT token.

  • After enabling Authentication, Authorization needs to be enabled in application.yml.

  • For Authentication, we can use LDAP, custom IdentityProvider, but for Authorization, we have to depend on the Camunda Authorization model, and that is maintained within Camunda local DB. So if we use a custom mechanism for Authentication where the user comes from an outside system and is authenticated by the outside system, then we need some way to map the authenticated user to an existing user in Camunda to make use of the Authorization model available within Camunda.

Basic Terms

Here are basic terms before understanding Camunda Authorization Model:

User

A user refers to a human individual.

Group

A group is any custom-defined "bundle" of users sharing some usage-relevant attributes.

Tenant

A tenant can represent a customer or a department of customers. Despite the fact that they share resources, tenants aren't aware of each other, and their data is kept totally secured from other tenants.

Memberships

  • A user can be a member of one or more groups.

  • Users and groups can be attached to one or more tenants.

Camunda Authorization

Permissions and restrictions for specific users or groups to access resources within Camunda (such as e.g. process definitions, tenants, process instances) are called authorizations. Because they relate users and groups to Camunda specific resources, they must of course always be managed in a Camunda specific way and be contained in the Camunda database.

Camunda comes with an AuthorizationService API (Java or REST), allowing the management of such Authorizations, and also ships with a dedicated Admin application to manage them through a web interface.

From the web apps perspective, it may look like that user task is assigned to a particular user and only he can complete it. But from the Rest API perspective, Authorization means having the right permission with the user who is calling the Rest API.

Using Camunda Authorization Model with Custom Authentication

The core of the Camunda engine treats users, groups, and tenants as simple text strings. Therefore you can e.g. do things like assigning a user task to a group of people by directly referencing the group in your BPMN file. In the case of custom Authentication, the authenticated user will come from an outside system and not necessarily exist in Camunda DB. But we can add a security filter that will map the Authenticated user to any existing user, group, and tenant in Camunda, and then that user will get the permissions according to the access permissions as per the mapped user/group/tenant.

Multi-Tenancy in Camunda

Multi-Tenancy can be achieved in two different ways. One way is to use one process engine per tenant. The other way is to use just one process engine and associate the data with tenant identifiers. We will mostly follow the one-process engine with tenant identifiers. Even in the case of the one-process engine, there are two deployment ways for process definitions.

Deploy Different Definitions for each Tenant

To deploy definitions for a tenant, the tenant identifier has to be set on the deployment. The given identifier is propagated to all definitions of the deployment so that they belong to the tenant.

If no tenant identifier is set then the deployment and its definitions belong to all tenants. In this case, all tenants can access the deployment and the definitions.

Shared Definitions for all Tenants

In the section “Deploy Definitions for a Tenant” it is explained how to deploy a Process Definition for a particular tenant. The result is that the definition is only visible to the tenant for whom it was deployed but not to other tenants. This is useful if tenants have different processes. However, there are many situations where all tenants should share the same definitions. In such situations, it is desirable to deploy a definition only once, in a way that it is visible to all tenants. Then, when a new instance is created by a particular tenant, it should be only visible to that tenant. This can be achieved by a usage pattern we call “Shared Definitions”.

When starting a new process instance, the tenant id of the process definition is propagated to the process instance. Shared resources do not have a tenant id which means that no tenant id is propagated automatically. To have the tenant id of the user who starts the process instances assigned to the process instance, an implementation of the TenantIdProvider SPI needs to be provided.

The TenantIdProvider receives a callback when an instance of a process definition is created. It can then assign a tenant id to the newly created instance.

Custom Authentication and Authorization with Admin-svc

We can use AadOauth annotations on the Application class to enable the custom Authentication. Once we add security annotations the Camunda REST APIs will need Bearer Token. But the user will not be authorized because those users do not exist in Camunda DB. To fix the Authorization issue we need to map the user to some existing Camunda Group and Tenant so that they get the right permissions.

  • For Camunda Group we can create a Group with required permissions in Camunda, and then use the same string to create a functional entitlement in DNAHQ-UI. Add this functional entitlement to a user, so that once it reaches Camunda then that functional entitlement will be fetched and set as a Camunda Group for the user.

  • For the granular level of authorization, we can create the same user in Camunda DB and give it the required permissions. Finally, the authenticated user will get the access permissions based on the mapped user, mapped group, and mapped tenant in the Camunda DB.

  • Camunda provides CRUD Rest endpoints for Authorization, User, Group, and Tenant. These Rest APIs can be used to create any required user and group in Camunda.

POCs

Below things are verified as POC using ce-svc-workflow-engine repo.

Authorization by mapping to Camunda Group

Created camunda-admin functional entitlement, and assigned it to a role camunda_admin. Assigned camunda_admin to e2etester11 user. In the security filter set the functional entitlements as Camunda Groups to the authenticated user. Now using the JWT token for starting a process and completing the user tasks works as it gets the Authorization as the user belongs to the camunda-admin group.

Tried the same with a custom demo group and custom authorizations.

Authorization by mapping to Camunda Tenant

Assigned an existing tenant to the authenticated user in the security filter, and it gets the Authorization as the user belongs to the tenant.

Shared Definitions for all Tenants

Tried this with a TenantIdProvider. In this case, the process is deployed without providing a tenant, but creating the process instance gets the Authenticated user’s tenant id and sets it through the security filter. Once the process instance is created and associated with a tenant, the Authorization works in the context of the associated tenant id.

If we use a TenantIdProvider then all the process instances will get created with the Authenticated user’s tenant id. Even if the process definition is deployed with some tenant id. Suppose a deployment is done for tenant1 and then that process is created using a user logged in with tenant2 then the process instance will be created with tenant2.

Note: In all the above examples, the user e2etester11 does not exist in the Camunda DB. Only group, tenant, authorizations, and group/tenant memberships were created.

Open Questions/Concerns

  • Camunda Web apps do not work once after enabling custom Authentication. Another known issue with web apps is that it does not work when we deploy 2 instances of the Camunda embedded engine.

  • We cannot sync all the users and tenants from admin-svc to camunda, so when and from where create the required users and tenants in Camunda.

  • User management in Camunda Database when camunda Authorization app is not working? Programatically we can use Camunda REST API to create users and groups but we should have some UI like DNAHQ UI to manage the users in Camunda DB. This UI might be helpful to sync the user between admin-svc and Camunda.

  • How to access DnaAuthorizationContext in security filter to retrieve required info like the user, functional entitlements, and tenants so that it can be mapped to existing entities in Camunda.

  • Authorization can be set for a particular resource id like an authorization can be created just for a process id. We need to decide how granular authorizations we want to create. Do we want a group in Camunda with read/write access to all processes or to a specific process?

Possible Solution

We can use DNA HQ UI can be used to manage users, groups, and tenants in the Camunda. To make it possible we have to create a rest-client for the embedded-camunda-engine service, similar to orchestration-client and admin-client. This client can be used to make calls to the Camunda engine. One possible thing that could be done is that whenever a group in Camunda and entitlement in admin-svc with the same string is required then we can call both APIs to create them together in a transaction from the DNAHQ UI.
