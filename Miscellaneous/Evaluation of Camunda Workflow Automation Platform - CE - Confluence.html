Deployment Models

Embedded Process Engine

In this case, the process engine is added as an application library to a custom application. This way, the process engine can easily be started and stopped with the application lifecycle. It is possible to run multiple embedded process engines on top of a shared database.

Container-Managed Process Engine

In this case, the process engine is started inside the runtime container (Servlet Container, Application Server). The process engine is provided as a container service and can be shared by all applications deployed inside the container or can be a Standalone Remote Process Engine Server.

Clustering Model

In order to provide scale-up or fail-over capabilities, the process engine can be distributed to different nodes in a cluster. Each process engine instance must then connect to a shared database.

Workflow Designing Tools

Camunda Modeler

Camunda Modeler is a desktop application for modeling BPMN workflows and DMN decisions. It’s user-friendly, allowing multiple developers to work together on the same diagrams.

Camunda Modeler supports BPMN 2.0 and DMN 1.3 (including decision tables and decision requirements diagrams).

Cawemo

Cawemo is a web tool for collaboratively Model BPMN Processes

Deploying Workflow

  • The workflow designed using Camunda Modeler can be deployed directly to Camunda engine using Camunda Modeler by calling rest API

  • Or save workflow designed using Camunda Modeler as .bpmn file and deploy from Camunda web UI

  • Or save workflow designed using Camunda Modeler as .bpmn file and put it in the resources folder, in case of Camunda engine running as Spring Boot embedded service. The process file will get automatically deployed with the start of the Spring Boot service.

Tools for Listing, Starting, and Monitoring Workflow

Cockpit

With Camunda Cockpit you can view all deployed workflows and decisions in production.

Tasklist

The Camunda Tasklist is a ready-to-use web application that allows end-users to work on the tasks assigned to them. It provides additional visibility when using the Camunda Workflow Engine for human task management.

REST API

The REST API allows you to use the process engine from a remote application or a JavaScript application. Camunda provides proper documentation of the REST APIs. Following things are tried using REST API through Postman client

  • Deploying a workflow

  • Starting a workflow

  • Completing a user task

Admin web application

Admin, accessible via http://hostname/camunda/app/admin/ is an application that allows you to configure users and groups via the engine’s Identity Service and authorizations via the engine’s Authorization Service. Furthermore, you can connect Camunda Admin to your LDAP system.

Task Delegation to external services

There are mainly 2 types of tasks: User Task and Service Task.

User Task needs to be assigned to a user/group and later performed by the assigned user/group. Tasklist web application can be used by a user to see the assigned tasks.

Service Task needs to be performed by some code. There are mainly 2 useful ways to delegate the service task.

  • Java Class: In this case, the task will use the Java Class Implementation type and will be attached to a class name. The provided class should implement JavaDelegate class and should run with the Camunda engine where the workflow is deployed.

  • External: In this case, the task will use the External Implementation type and provide a Topic name. There should be an external service subscribed to the provided Topic name so that it can poll and fetch the task coming to that topic. The external service can be created in any language. Camunda has provided clients with Java and JavaScript languages. There are clients available in other languages but not supported by Camunda.

Securing Camunda

Authentication

The REST API ships with an implementation of HTTP Basic Authentication. By default, it is switched off, but can be activated by adding a servlet filter. HTTP Basic Authentication is tested as part of POC. After enabling the HTTP Basic Authentication, the Camunda REST APIs need username/password.

If you require an authentication mechanism other than HTTP Basic Auth, you need to implement your own AuthenticationFilter.

The difficulty with Single Sign-on (SSO) is that it always works a bit differently depending on your exact environment and SSO framework used. Therefore, Camunda only provides the hooks for plugging in the exact mechanism that fits your needs. The key part that you need to touch concerns the authentication filters of Camunda’s web applications and the REST API (see illustration above). The idea is to exchange or modify the AuthenticationFilter defined inside the WEB-INF/web.xml file of Camunda’s web applications (or the REST API) and hook in your custom SSO provider. That means that your filter implementation circumvents the redirect to Camunda’s login page by telling Camunda directly who is currently logged in.

From Camunda 7.9 on, it is much easier to implement SSO by making use of the ContainerBasedAuthenticationFilter. This means that you do not need to replace the existing AuthenticationFilter by a custom one anymore, but you only need to add the ContainerBasedAuthenticationFilter that ships with the product and implement a custom AuthenticationProvider if required.

Authorization

Permissions and restrictions for specific users or groups to access resources within Camunda (such as e.g. process definitions, tenants, process instances) are called authorizations. Because they relate users and groups to Camunda specific resources, they must of course always be managed in a Camunda specific way and be contained in the Camunda database.

Camunda comes with an AuthorizationService API (Java or REST), allowing to manage such Authorizations and also ships with a dedicated Admin application to manage them through a web interface.

 

Recommendations

Deployment Model

From a development perspective using an Embedded Camunda engine with Spring Boot is recommended. It will provide the following benefits:

  • Controlled by developers

  • Easily configurable for different databases, and properties.

  • Can use vault/consul properties like other services for DB username, password.

  • Easy deployments to different environments with DevOps pipeline

  • Easy upgrade of the Camunda engine version. It can be done by just updating the library version in the Gradle properties file.

Workflow Deployment

Camunda modeler should be used to create the workflow visually and the bpmn file should be stored with the embedded engine. It will provide the following benefits:

  • Workflow definitions will be persisted in the repository.

  • Any change in a workflow will get deployed automatically with the service restart.

  • workflow definition will be rolled out easily to STAGE and PROD environments.

  • As the workflow will be deployed automatically, so fewer chances of manual error like providing different definitions to different environments in case of manual deployment of workflow.

Task Delegation

Java Delegate Class within Camunda engine should be used. It will provide the following benefits:

  • The task definition will be available with the engine service, so everything related to a workflow will be in one service. There will not be any code change in service to support a Task of the workflow. A service can have only the core logic/implementation required for the service and it should provide REST APIs and REST clients to be used by the Java Delegate class.

  • Logging will be available at one place for the whole workflow execution.

  • Testing/Debugging a workflow will be easy whole developing workflow. It is not possible in orchestration service to test a workflow while developing if multiple services are participating in a workflow.

  • Less dependency on other services, from a workflow definition perspective. Considering there is no bug in service then changes will be done in only embedded engine service. And it will easily be deployed without waiting for other services. Currently for the Orchestration service if there is any bug in task implementation by a service then we have to wait till the time it is fixed and deployed.

Concerns/Questions

  • Camunda Platform Initializr has provided the template for the only maven, so it should be tried with Gradle.

  • Camunda version 7.15.0 uses Spring Boot version 2.4.3, whereas our all services and common components use Spring Boot version 2.2.11.RELEASE. So not sure if that will cause any issue in case vault common components are used.

  • Camunda packages come with in-memory H2 Database support, what about supporting different Databases?

  • How will it work when multiple instances of the embedded engine will run?

  • What about the UI in case of multiple instances?

  • How easily we can integrate it with other common vault components like audit, notification?

  • Can we use Java API (calling Camunda engine’s method directly) in a wrapper Service, instead, using REST API from another service?

Findings/Answers

Embedded Camunda with Gradle

As part of POC a Gradle project (Java Service created by the DevOps using template java service) is used to run the embedded Camunda engine. repo link: https://github.com/eagleinvsys/ce-svc-workflow-engine - Connect to preview

Spring Boot Version

Our services and components use Spring Boot version 2.2.11.RELEASE and Camunda latest version 7.15.0 uses Spring Boot version 2.4.3. As part of POC spring-boot service with embedded Camunda engine is tried for Camunda 7.13.0 as it is compatible with Spring Boot version 2.2.5.RELEASE and it is working fine.

MySQL Database

As part of POC, a PERCONA DB is used and configured in application.yml for the embedded camunda service and it is working fine.

Multiple Instances of Embedded Camunda Engine Service

As part of POC, 2 instances are deployed with a shared Percona DB schema. The REST APIs work properly, but the Camunda web applications are not working.

Integration with Vault Components

As part of POC, common annotations are used on the Application class and then all the Camunda REST APIs got secured. But the Web applications were not reachable.

Java API vs. REST API

As per Camunda docs: if you are working with an embedded engine using spring-boot (with the camunda spring boot starter), you can use @Autowired to inject the ProcessEngine interface in your bean. There, you will get access to some services.

In the case of Java API calls, and assuming we will interact with some Vault components or services, then we have to degrade the Camunda version to match the Spring Boot version. But if we leave the embedded Camunda engine service running separately and communicate with it through REST API then it can use the latest Camunda version and can also be upgraded in the future.
