Camunda Authorization Basic Principles

Identities

  • Camunda Platform distinguishes between two types of identities: users and groups. Authorizations can either range over all users (userId = ANY), an individual user, or a group of users.

Permissions

  • A Permission defines the way an identity is allowed to interact with a certain resource. The basic permissions available in the engine are:

    • None

    • All

    • Read

    • Update

    • Create

    • Delete

    • Access

Resources

  • Resources are the entities the user interacts with. The following resources are available:

Authorizations

An Authorization assigns a set of Permissions to an identity to interact with a given Resource.

Camunda Admin Group

By default, Camunda creates a camunda-admin group and assigns all Permissions to it to interact with all Resources.

Managing Camunda Authorization using Vault Entitlements

  • Camunda can manage the Authorizations using Camunda groups. If an Authenticated user comes with a Camunda Group name then he will be given the access permissions as per the Group name.

  • We can create different Camunda Groups for every Camunda Resource and give it ALL permissions.

  • Now a user needs to have the right Camunda Group name or set of Camunda Group names to get the required access permissions for Camunda Resources in the Camunda engine.

Creation of Camunda Groups and Vault Functional Entitlements

  • A scheduled job will be used in the svc-workflow-engine service to create all the required Camunda Groups and Authorizations.

  • Required Functional entitlements will be created using @RegisterPolicy Annotation in the svc-workflow-engine service.

  • This way the application will be ready with all the required Camunda Groups and mapped Functional entitlements in Vault. We can create required roles from Camunda Functional Entitlements and assign them to the required user.

E2E Test Cases

Roles And Users required for e2e tests

Roles

  • A new role “WORKFLOW_ENGINE_PROCESS_ADMIN" will be created with following entitlements:

    • wfengine.deployment.all

    • wfengine.process.definition.all

    • wfengine.process.instance.all

    • wfengine.task.all

Users

  • 2 Users will be used for e2e tests:

    • e2etester11 with “WORKFLOW_ENGINE_PROCESS_ADMIN" role

    • e2etester12 without “WORKFLOW_ENGINE_PROCESS_ADMIN" role
